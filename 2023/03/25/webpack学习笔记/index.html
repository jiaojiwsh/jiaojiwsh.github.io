# 一. webpack 介绍

# 1.1 webpack 介绍

webpack 是一个打包模块化 javascript 的工具，在 webpack 里一切文件皆模块，通过 Plugin 注入钩子，最后输出由多个模块组合而成的文件。webpack 专注于构建模块化项目

一切文件：Javascript、css 、scss、图片、模板，在 webpack 眼中都是一个个模块，这样的好处是能清晰描述出各个模块之间的依赖关系，以方便 webpack 对模块进行组合和打包，经过 webpack 的处理，最终会输出浏览器能使用的静态资源。他的目的就是把有依赖关系的各种文件打包成一系列的静态资源.

webpack 的优点：

  1. 专注于处理模块化的项目，能做到开箱即用一步到位
  2. 通过 Plugin 扩展，完整好用又不失灵活
  3. 使用场景不仅限于 Web 开发
  4. 社区庞大活跃，经常引入紧跟时代发展的新特性，能为大多数场景找到已有的开源扩展
  5. 良好的开发体验

# 1.2 安装

先装好 node 和 npm，因为 webpack 是一个基于 node 的项目。然后

  • -S 安装到上线环境
  • -D 安装到开发环境
  • -g 安装到全局
  • webpack-cli 用于在命令行中运行 webpack，cli 即命令行接口 (Command Line Interface)

测试:

webpack 零配置：创建 src 文件–> 创建 indexjs 文件–> 运行 webpack (对项目进行打包)

  1. 开发环境 (development): 开发环境是程序猿们专门用于开发的服务器，配置可以比较随意，为了开发调试方便，一般打开全部错误报告。
  2. 测试环境：一般是克隆一份生产环境的配置，一个程序在测试环境工作不正常，那么肯定不能把它发布到生产机上。
  3. 生产环境 (production): 是指正式提供对外服务的，一般会关掉错误报告，打开错误日志。

# 二. webpack 的基本配置

需要在项目的根目录下创建一个文件 ==webpack.config.js

四个核心概念

  • 入口（entry）
  • 出口（output）
  • loader
  • 插件（plugins）

# 1. 基本配置 - 入口 (entry)

入口起点 (entry point) 指示 webpack 应该使用哪个模块，来作为构建其内部依赖图的开始。进入入口起点后，webpack 会找出有哪些模块和库是入口起点依赖的

每个依赖项随即被处理，最后输出到称之为 bundles 的文件中。

可以通过在 webpack 配置中配置 entry 属性，来指定一个入口起点（或多个入口起点），== 默认值为./src

# 2 基本配置 - 出口（output）

output 属性告诉 webpack 在哪里输出它所创建的 bundles，以及如何命名这些文件，== 默认值为./dist,== 基本上，就整个应用程序结构，都会被编译到你指定的输出路径文件中。你可以通过在配置中指定一个 output 字段，来配置这些处理过程

# 4. 基本配置 - loader（加载器）

loader 让 webpack 能够去处理哪些非 javascript 文件（webpack 自身只理解 javascript）。loader 可以将所有类型的文件转换为 webpack 能够处理的有效模块，然后就可以利用 webpack 的打包能力，对它们进行处理

在更高层面，在 webpack 的配置中 loader 有两个目标：

  1. text 属性，用于标识出应该被对应的 loader 进行转换的某个或某些文件。
  2. use 属性，表示进行转换时，应该使用哪个 loader。

webpack.config.js

以上配置中，对一个单独的 module 对象定义了 rules 属性，里面包含两个必须属性： test 和 use 。这告诉 webpack 编译器 (compiler) 如下信息：

“嘿，webpack 编译器，当你碰到「在 require() / import 语句中被解析为 ‘.txt’ 的路径」时，在你对它打包之前，先使用 raw-loader 转换一下。”

重要的是要记得，在 webpack 配置中定义 loader 时，要定义在 module.rules 中，而不是 rules 。然而，在定义错误时 webpack 会给出严重的警告。为了使你受益于此，如果没有按照正确方式去做，webpack 会 “给出严重的警告”

# 5. 基本配置 - 插件 (plugins)

loader 被用于转换某些类型的模块，而插件则可以用于执行范围更广的任务。插件的范围包括，从打包优化和压缩，一直到重新定义环境中的变量

想要使用一个插件，只需 require () 它，然后添加到 plugins 数组中。多数插件可以通过选项 (option) 自定义。也可以在一个配置文件中因为不同目的而多次使用同一个插件，这时需要通过使用 new 操作符来创建它的一个实例。

# 6. webpack 打包流程

Webpack 是一个现代的 JavaScript 应用程序打包工具。它是一个经过优化的、可扩展的构建工具，能够将各种文件 (JavaScript、CSS 图片等) 打包到单个 JavaScript 文件中。
Webpack 的打包过程结构非常清晰，可以分为以下几个步骤：

# 6.1 入口配置

从 webpack 的配置文件中，开始读取项目文件的入口文件。webpack 会分析每个文件的依赖关系，然后按照这些依赖关系进行打包操作。

# 6.2 Loaders

webpack 中的 Loaders 用于处理各种不同类型的文件。例如，webpack 的默认加载器能够处理 JavaScript 文件和 JSON 文件，但是如果我们需要使用其他类型的文件，例如 CSS 文件、图片文件、字体文件等，就需要使用相应的加载器。安装好相关的加载器后，我们可以在配置文件中指定相应的加载器。

# 6.3 Plugins

webpack 的插件是一种用于增强 webpack 功能的集成式组件。公全 屏使用 webpack 时，我们可以通过安装不同的插件，增加额外的功能例如压缩代码、提取公共模块、图片压缩、自动生成 HTML 文件等。大 webpack 的配置文件中，我们需要指定我们要使用的插件。

# 6.4 输出文件

在第一步配置好入口文件、第二步配置好加载器和第三步配置好插件后，webpack 会自动地将所有的文件打包为一个或多个 JavaScript 文件。我们需要在输出文件中指定打包后文件的存储位置文件名和文件格式。
上述四个步骤构成了 webpack 打包的全过程。总的来说 webpack 是一个非常方便的 JavaScript 打包工具，能够大大提高我们的开发效率。

# 三。模块化

# 1. 介绍

模块化是一种思想，是一种把复杂系统分解为更好的可管理模块的方式。把复杂的问题分解成相对独立的模块，降低程序的复杂性，提高代码的复用性。把具有一定独立功能的代码放在一个单独的文件中，每一个文件就是一个模块。
